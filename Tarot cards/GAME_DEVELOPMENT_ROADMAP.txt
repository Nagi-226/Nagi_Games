塔罗牌占卜系统 - 游戏化开发路线图
==========================================
编写日期: 2026-04-09
编写视角: 游戏行业资深开发 (10年+)
基于项目现状 v1.0 的分析

==========================================
一、当前项目技术诊断
==========================================

【技术栈】
  前端: React 18 + Vite + Framer Motion + React Router 6
  后端: Express + sql.js (内存SQLite)
  数据: 78张塔罗牌 + 5种牌阵 (硬编码初始化)

【核心问题清单】(按严重程度排序)

  1. [致命] 无用户系统
     - 无登录/注册，所有数据无归属
     - 成就、等级、社交等游戏化功能全部无法落地
     - 这是整个游戏化体系的地基，必须最先做

  2. [致命] sql.js 性能瓶颈
     - 纯内存数据库，服务重启数据丢失
     - 不支持并发写入，多用户场景会出问题
     - 78张牌数据每次启动重新插入，浪费资源

  3. [严重] 无音效系统
     - 塔罗占卜的核心体验是仪式感，音效占仪式感50%以上
     - 没有听觉反馈，用户体验大打折扣

  4. [严重] 卡牌无图无视觉表现
     - 只有文字，用户无法建立与卡牌的情感连接
     - 在游戏行业，美术资源是核心资产

  5. [中等] 抽牌逻辑有缺陷
     - initializeDeck 取的是洗牌后前N张，不是从78张中随机抽取
     - 用户点击牌堆中的牌才能抽取，但牌堆就是那前N张，等于没有随机性
     - 正确做法：展示78张牌背，用户点击时从未抽牌池中随机选一张

  6. [中等] 无状态管理
     - 纯 useState，组件间数据流混乱
     - 后续游戏化系统(成就/等级/任务)需要全局状态

  7. [轻微] 前端用了CRA的环境变量写法
     - process.env.REACT_APP_* 在 Vite 中不存在
     - 已修复: 改为使用 Vite proxy 的相对路径 '/api'

  8. [轻微] 无错误边界
     - 任何组件报错会导致整个白屏(就是之前的bug)
     - 需要 React Error Boundary 做降级展示


==========================================
二、重构后的迭代优先级 (与原规划对比)
==========================================

原规划的问题：优先级不够准确，比如新手引导排在音效前面，
用户系统推迟到了第二阶段，这是游戏行业的大忌。
以下是基于游戏开发经验的调整建议：

--------------------------------------------------
阶段零: 基础设施 (2-3周) [原规划中没有]
--------------------------------------------------

  必须先做的地基工作，否则后续全部建在沙子上：

  0.1 用户系统
      - 注册/登录 (邮箱 + 社交登录)
      - JWT 认证中间件
      - 用户表 (id, nickname, avatar, created_at)
      - 前端登录/注册页面
      优先级: 最高 | 工期: 5天
      理由: 没有用户系统，成就/等级/收藏全都是空中楼阁

  0.2 数据库迁移 sql.js -> better-sqlite3
      - 文件持久化存储
      - 添加索引
      - 连接池
      - 数据迁移脚本
      优先级: 最高 | 工期: 2天
      理由: sql.js 重启丢数据，线上不可接受

  0.3 React Error Boundary
      - 全局错误捕获
      - 降级UI展示
      - 错误上报
      优先级: 高 | 工期: 0.5天
      理由: 防止一个组件报错导致整站白屏

  0.4 状态管理引入 (Zustand)
      - 全局用户状态
      - 卡牌数据缓存
      - 音效设置状态
      优先级: 高 | 工期: 2天
      理由: 后续游戏化系统需要全局状态支撑

  0.5 修复抽牌逻辑
      - 展示78张牌背
      - 点击从未抽牌池随机抽取
      - 加入抽牌动画
      优先级: 高 | 工期: 1天
      理由: 核心玩法不能有逻辑缺陷

--------------------------------------------------
阶段一: 核心体验 (1-2个月)
--------------------------------------------------

  1.1 音效系统
      - 背景音乐 (首页神秘氛围 + 占卜仪式音乐)
      - 交互音效 (翻牌声、按钮点击、成就解锁)
      - 环境音效 (蜡烛声、风铃声)
      - AudioManager 全局管理器
      - 音量控制 UI
      优先级: 最高 | 工期: 1周
      理由: 塔罗占卜的灵魂是仪式感，音效占仪式感的一半

  1.2 卡牌视觉升级
      - 78张卡牌美术资源 (可AI生成 + 人工微调)
      - 正逆位不同视觉表现
      - 悬停光晕效果
      - 金边装饰
      优先级: 最高 | 工期: 2周 (资源制作为主)
      理由: 视觉是用户第一印象，纯文字无法建立情感连接

  1.3 粒子特效系统
      - 首页星尘粒子
      - 占卜页蜡烛火焰
      - 翻牌光芒扩散
      - 完成占卜星光汇聚
      优先级: 高 | 工期: 1周
      理由: 配合音效增强仪式感

  1.4 新手引导
      - 首次访问3-5步引导流程
      - 完成引导解锁特殊牌阵
      - 帮助中心
      优先级: 中 | 工期: 3天
      理由: 降低学习成本，但不是最紧迫的

  1.5 性能优化
      - 路由懒加载
      - 图片懒加载 + WebP
      - 骨架屏 Loading
      - API 响应缓存
      优先级: 中 | 工期: 1周
      理由: 随内容增多会越来越需要

  1.6 移动端适配 + PWA
      - 响应式优化
      - 触摸手势
      - 离线访问
      - 添加到主屏幕
      优先级: 中 | 工期: 1周
      理由: 塔罗牌天然适合移动端场景

--------------------------------------------------
阶段二: 游戏化体系 (2-3个月)
--------------------------------------------------

  2.1 成就系统
      - 占卜成就 (初次占卜 / 达人 / 牌阵大师 / 每日一占)
      - 探索成就 (牌库收藏家 / 深度解读)
      - 特殊成就 (幸运儿 / 完美平衡 / 大阿卡那之路)
      - 成就墙 + 解锁动画 + 音效
      优先级: 最高 | 工期: 1.5周
      理由: 游戏化的核心粘性机制

  2.2 等级与经验系统
      - 10个等级: 新手 -> 见习 -> 学徒 -> 占卜师 -> 大师 -> 贤者 -> 导师 -> 先知 -> 大贤者 -> 至尊占卜师
      - 经验来源: 占卜/查看牌意/解锁成就/每日签到/分享
      - 等级特权: 解锁牌阵/主题
      - 升级动画 + 庆祝特效
      优先级: 最高 | 工期: 1.5周
      理由: 与成就系统配合，构建成长曲线

  2.3 每日任务系统
      - 每日占卜 / 每日三牌 / 每日探索 / 每日分享
      - 0点刷新 + 进度条 + 一键领取
      优先级: 高 | 工期: 1周
      理由: 培养日活习惯的关键机制

  2.4 收藏与个性化
      - 收藏塔罗牌/占卜记录
      - 多套UI主题 (神秘紫/星空蓝/火焰红/森林绿)
      - 卡牌背面样式选择
      - 个人中心
      优先级: 中 | 工期: 1.5周
      理由: 增强归属感，为后续商业化铺路

  2.5 数据统计与可视化
      - 占卜趋势图 / 正逆位比例 / 牌阵使用频率
      - 最常抽到的牌 TOP10
      - 全局统计 (热门牌阵/热门塔罗牌)
      优先级: 中 | 工期: 1周
      理由: 数据驱动迭代，同时增强用户粘性

--------------------------------------------------
阶段三: 社交与商业化 (3-4个月)
--------------------------------------------------

  3.1 分享系统
      - 生成精美分享图片
      - 微信/微博/QQ 分享
      - 分享获经验奖励
      优先级: 高 | 工期: 1.5周

  3.2 社区系统
      - 占卜结果分享墙
      - 点赞/评论/关注
      - 内容审核机制
      优先级: 中 | 工期: 2周

  3.3 AI 增强功能
      - GPT 驱动的深度牌意解读
      - AI 占卜师对话
      - 智能牌阵推荐
      优先级: 高 | 工期: 2周
      理由: 差异化竞争力，但需控制 API 成本

  3.4 商业化系统
      - 付费主题包 / 高级牌阵 / AI 深度解读
      - 会员体系 (月度/年度)
      - 激励视频广告
      优先级: 中 | 工期: 2周
      理由: 确保可持续运营，但不能影响免费用户体验

  3.5 多语言国际化
      - 简中/繁中/英/日
      - UI + 塔罗牌数据 + 帮助文档
      优先级: 低 | 工期: 2周
      理由: 扩大市场，但需先验证国内模型


==========================================
三、技术架构演进路线
==========================================

【阶段零后】
  server/src/
  ├── controllers/     # 控制器层
  ├── services/        # 服务层
  ├── repositories/    # 数据访问层
  ├── middlewares/     # 认证/日志/错误处理
  ├── models/          # 数据模型
  └── config/          # 配置

  client/src/
  ├── stores/          # Zustand 状态管理
  ├── hooks/           # 自定义 Hooks
  ├── components/      # 通用组件
  ├── pages/           # 页面组件
  ├── services/        # API 服务
  ├── utils/           # 工具函数
  └── assets/          # 静态资源(卡牌图/音效)

【阶段二后 - TypeScript 迁移】
  - 新功能直接用 TS 编写
  - 核心模块逐步迁移
  - 类型定义: TarotCard, Spread, Reading, User, Achievement...

五、与原 OPTIMIZATION_PLAN.md 的主要差异
==========================================

  1. 新增"阶段零"：用户系统和数据库迁移必须先做
     原规划把用户系统隐含在阶段二的等级/成就中，
     但没有用户系统，这些功能无法实现

  2. 音效和视觉升级优先于新手引导
     原规划中新手引导排在音效前面，
     但仪式感(音效+视觉)才是塔罗产品的核心差异化

  3. 抽牌逻辑修复提上日程
     原规划未提及当前抽牌逻辑的缺陷

  4. Error Boundary 作为基础设施
     原规划未提及错误边界，但白屏问题说明这是刚需

  5. 预算建议调整
     原规划50.5万/年偏保守，建议：
     - 优先投入美术资源（卡牌图是核心资产）
     - AI生成卡牌图可大幅降低成本（5000元以内）
     - 音效可用开源/AI生成，初期成本可控（1000元以内）
